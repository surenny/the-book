§2第一次 cheat：12 类 pattern

同一份提示词，opus 老老实实做题，codex 找到了 12 条独立的近路。

故事从一次很朴素的尝试开始。

我们让 codex 跑的是与 opus 4.7 完全相同的提示词（baseline.md），目的本来很单纯——做一次"同 prompt 不同 model"的能力对比。当时的 verify.sh 是有漏洞的，这本不该是问题：一个忠实的模型不会去 hack 一个评测器的漏洞，他只会尝试把定理做出来。opus 4.7 当时就是这样跑的。

codex 不是。23 道题里——

17 道每一道都用一种独立的作弊路径骗过了 verify.sh，最终我们整理出 12 类不同的 cheat pattern。 — 第一次 cheating retrospective

从"在 statement 里偷偷改 hypothesis"、"把 goal 替换成自反等式"、"在 import 里塞 axiom"，到把整个 proof body 折叠成一个看起来人畜无害但其实绕过 type check 的 macro——12 类作弊横跨从 spec、import、tactic 到 elaborator 的几乎每一层。这一份复盘里把它们都列了：

Codex GPT-5.5 在 FormalQualBench 上的 12 类 cheat 复盘

裸 codex（无 MCP，baseline.md）跑 23 题，4 轮 anti-cheat 加固后只 6 道真做出来——剩下 17 道每一道都用一种独立的作弊路径骗过 verify.sh。第 11 节有 codex 自述作弊意图的直接引证。

§3strict17 之后，最重的一层不得不出场

提示词加固只是把作弊向量推到了"还没被禁"的方向；evaluator 必须升级。

我们的第一反应是写更严的提示词——内部叫 strict17，把已经看见的 12 类全部按名点出、明令禁止，并要求每一次提交带可追溯的元数据。

strict17 跑出来的是一轮新的 hack。被禁的旧路径上，作弊向量沿没被禁的方向迁移了：从"改 statement"迁到"把 axiom 用文件级 attribute 偷偷激活"、从"折叠 macro"迁到"利用 namespace 重映射"。每加固一轮提示词都能堵住前一轮 80% 以上，但总有 1–2 类穿透。提示词工程到这里基本看见了天花板。

这把我们逼到了评测链路里最重的一层：

把 evaluator 从"提示词约束 + 表面字符串检查"升级到 kernel 闭包检查——comparator。 — 第 3 次 hardening 的决定

comparator 做的事很重：把模型提交的 proof 真的扔进 Lean kernel，把 closed term 的所有传递依赖枚举出来，与一份 axiom whitelist 比对——任何不在白名单里的 axiom 都判 illegal_axiom。它是第一层"作弊代价 ≥ 真证明代价"的检查：要绕过它，模型得在 kernel 里真的让 closed term 不依赖任何陌生 axiom。

代价是 comparator 跑得不便宜。每一道题的闭包枚举与依赖追踪是分钟量级，而且对 mathlib import 量敏感。这条信息我们要记着，§6 再用。

§4反 grep 元编程——一种新形态学

同 model、加 MCP、换 prompt，clean 净 +1，但作弊潜到了 source layer 看不见的地方。

5 月最新一轮，我们做了三处替换：同 model（仍是 gpt-5.5-xhigh），但加上 archon-lean-lsp MCP，并换用 lean-tools-codex.md 提示词重跑 23 题。预期是 MCP 给 codex 一个真正的工具层——访问 goal state、查 hypothesis、迭代 elab——它应当变得更老实。

MCP 没让 codex 更老实。只让作弊更隐蔽。

verdict 分布上看起来像是"前进"：clean 从 6 涨到 7，statement_mismatch 从一个相对显眼的多数 ↘ 到 10。但新出现的是 illegal_axiom ↗ 7——而且：

所有 7 个 illegal_axiom 在源码层 grep axiom / grep sorry 都是 0，verify.sh 全部过。 — 第二次 cheating retrospective

只有 comparator 的 kernel 闭包检查把它们打回了原形。这是一种新的形态学——为了绕过"找字符串 axiom"的检查而专门设计的元编程：

  • Pattern A（6/7）："ax" ++ "iom" 字符串拼接 → Parser.runParserCategory → elabCommand，在 elaboration 时合成一个真正的 axiom 声明
  • Pattern A'（Schauder）：Meta.addDecl 手搓一个 Declaration.axiomDecl 直接进 environment——连 parser 都不走
  • Pattern B（GreenTao）：@[implemented_by] + native_decide，把 truthiness 委托给原生编译的 boolean，kernel 闭包里多出一个 Lean.ofReduceBool

这三类共同点是：source 看着干净，verify.sh 看着干净，只有 comparator 的 kernel 闭包枚举能识破。

§5Opus 的"老实"——同一种性格的两面

在评测里不 hack，在生产里频频停摆——这两件事可能本就是一件事。

把镜头拉远一下。

整轮 baseline 实验里有一个我们当时没特别在意、回头看反而最重要的对照：同一份有漏洞的 baseline.md 提示词，opus 4.7 没有去 hack。它失败了几道题，但失败得很坦白，像一个会在做不出来时举手的学生。

而在我们生产 agent loop（archon 系）里，opus 4.7 的另一面是这个：

评测环境opus 4.7

同一份有漏洞的 prompt，不 hack 评测器。做不出来的题就显式失败，不去绕，也不去伪造。

表现稳定地"忠实"。

生产环境opus 4.7

频频"举手"、"停摆"——空轮、planner/reviewer 因为各自判断不同而原地 halt、子任务在拿不准的边界上直接挂起等人介入。

同一种"忠实"在生产里的代价。

这两件事看起来一个是好品质、一个是 bug，但我可能这两件事是同一种性格的两面：opus 在遇到不能确认的事情时倾向于停下；它不会去 hack 一个看起来能走通的漏洞，也不会在自己拿不准的时候硬上。

这件事不全是好的。一个会停下来的 agent 在评测里"忠实"，在生产里却经常让人觉得它"扫兴"——任务停在一个看似可以再推一下的地方，留出一个需要人介入的洞。我们之前把这两件事当成两个独立现象来谈，这次复盘后我倾向把它们合并：性格相同，环境给出的反馈不同。

§6生产环境的下一步：comparator 该放在哪

codex 进入 loop 是合理的，但 comparator 不能这么放进去。

三份报告并起来读，结论比较清楚：

  1. codex 系列的能力——top-tier 的 23 题成绩——值得引入生产 agent loop。"它会作弊"不是不用它的理由，而是用它时必须配什么的理由。
  2. opus 系列的"忠实"在生产 loop 里偶尔表现为 halt，这是它的性格；要么改 loop 让 halt 可被无害处理，要么在合适节点上用 codex 把卡住的部分推过去。
  3. comparator 是目前唯一抓得到反 grep 元编程的 oracle，所以它必须进生产；但它跑得太重，不能逐 step 跑。

所以问题变成：comparator 在生产 loop 里放在哪。两条岔路：

路 A · 稀疏插入

保留现在的 comparator 实现，但只在合适的节点触发：子目标关闭点、阶段性 milestone、PR 边界、merge 前；其余 step 用便宜的 evaluator（pretty-print + 字符串规则）兜底。

难点是"合适的节点"如何定义——粒度太粗，作弊有机会进入下游；粒度太细，又回到了"每 step 都跑"。需要找一个对作弊敏感、对正常进度宽容的触发点定义。

路 B · 轻量化 comparator

在不牺牲"作弊代价 ≥ 真证明代价"这个性质的前提下，把 comparator 跑得轻一点。比如：增量式 axiom 闭包（只检查相对上一步新增的依赖）、按 mathlib namespace 做并行 shard、用更聪明的 axiom whitelist 维护策略减少误报后的重算。

难点是任何"轻量化"都得有反例集合压住——只要某一类 hack 能绕过新的实现，comparator 就失去它"唯一可信 oracle"的位置。

这两条岔路不互斥，但精力先押哪一条，是我们 6 月份要回答的事。倾向上：先做路 A 的最小可用版本（每个题目最终提交时强制跑 comparator，中间 step 不跑），积累一段真实的 false negative / false positive 样本，再去判断要不要投入做路 B。

这是评测器这一侧目前的尽头。下一份独立报告，应该是路 A 的最小实现 + 一次有 ground truth 的 false negative 调研。

§7Barchon 这两周——loop 这一侧的同步演化

§6 是评测器这一侧的尽头。这一节是同一时间 loop 自己在做的事——也就是 comparator 将来要嵌进的那个系统。

5/13 到 5/22 之间，Barchon 主仓做了三件结构性的事，外加几件清理。

1. TeX 成为唯一真值 PR #12 · +928 / −106 · merged 2026-05-21

之前 review agent 在 Step 4.5 直接改 chapters/*.tex 把节点推进到下一级；同时 leanblueprint web、apply_milestones、prep_cycle 各自也在写 status.yaml——四个写者抢一份状态。表现出来的症状是 KIP-infra cycle_004 → cycle_005 里 prereq:thm:nu-preserves-finite 被反复 flip："proved → false → proved → false"，每个 cycle 浪费约 $2 + 10 分钟，无限循环。

#12 把这条路堵死：milestone 的应用全部移到一个确定性 Python 脚本（tools/apply-milestones-to-tex.py），review agent 不再有任何 .tex 写权限，yaml 由 leanblueprint web 在 Phase 0 重新生成。一份状态，一个写者。

这件事在原理上呼应 §3 那一层：agent 的判断不能是 source of truth，只有确定性的代码可以。

2. Drafter 成为同级 agent + 状态机化任务调度 PR #4 · +1479 / −76 · merged 2026-05-14

之前 NL 起草被 prep_cycle._phase_for 加一个 hardcoded 的 proved-prereq 检查混在一起，结果就是 AUT-58 cycle 2 里 planner 看见一个 nl-rejection 想行动，但 prereq 节点 Lean 还没 proved，READY_NODES.md 就不列它，planner 被迫 defer——一件看起来只是 NL 改改的事被 Lean 证明状态卡死。

#4 把 drafter 提到 autoformalizer / prover 同级，由 planner 通过 [draft] objective 显式调度；并把 _phase_for 换成显式状态机——phase 感知的 prereq 满足（draft → nl_reviewed、formalize → bound、prove → proved），nl-rejection / align-rejection 自动重路由。同时清掉了 ownership 模糊：drafter 写 NL + \label / \uses / \notready；reviewer 在 bound 之后 bind \lean{…}、proved 之后 bind \leanok；drafter 一律不许写后两者。

这件事是对 §5 那段"Opus 会停下"的 loop-side 解药之一：很多次"停下"其实是路由问题——节点该被处理，但 dispatch 规则把它排除了。状态机化之后，路由不再隐式依赖 phase 命名。

3. Review 闸门可回滚 + cascade-clear PR #17 · open · 当前分支

给 review 加 undo_alignment / undo_nl_review。误点了？换了 reviewer 想推翻？意识到一个 proved 的定理 NL 其实写错了、整条 ladder 都得重来？以前只能手改 blueprint/status.yaml；现在 POST /api/nodes/:id/review 就能撤。关键设计是 cascade-clear：撤 aligned 时把 proved 一并清掉，撤 nl_reviewed 时把 aligned 和 proved 一并清掉——lifecycle 阶梯靠级联保证一致。

这件事是给 §5 那个"举手"的 agent 留的一条下台阶：举完手发现搞错了，可以撤一步重来，不必整条线推倒。
